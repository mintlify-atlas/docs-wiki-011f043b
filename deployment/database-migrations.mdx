Apache Fineract uses Liquibase to manage all database schema changes. The master
changelog is at
`fineract-provider/src/main/resources/db/changelog/db.changelog-master.xml` and
orchestrates two schema targets, the core tenant changeset sequence, and
per-module extensions. Migrations run automatically at application startup before
Spring Boot serves any traffic.

## Two Schema Targets

| Schema | Context Flag | Purpose |
|--------|-------------|---------|
| `fineract_tenants` | `tenant_store_db` | Tenant registry: one row per registered tenant, JDBC connection parameters |
| `fineract_default` (per-tenant) | `tenant_db` | All domain tables for a single tenant |

The master changelog selects the appropriate changeset tree using Liquibase
`context` attributes:

```xml
<include file="tenant-store/changelog-tenant-store.xml"
         context="tenant_store_db AND !initial_switch"/>
<include file="tenant/changelog-tenant.xml"
         context="tenant_db AND !initial_switch"/>
```

The `initial_switch` context is used for one-time migration from older Fineract
releases.

## Master Changelog

`fineract-provider/src/main/resources/db/changelog/db.changelog-master.xml`:

```xml
<databaseChangeLog ...>
    <!-- DB-specific property aliases -->
    <property name="current_date"     value="CURDATE()"          context="mysql"/>
    <property name="current_date"     value="CURRENT_DATE"       context="postgresql"/>
    <property name="uuid"             value="uuid()"             context="mysql"/>
    <property name="uuid"             value="uuid_generate_v4()" context="postgresql"/>

    <!-- Tenant store (fineract_tenants schema) -->
    <include file="tenant-store/changelog-tenant-store.xml"
             context="tenant_store_db AND !initial_switch"/>

    <!-- Core tenant schema (fineract_default) -->
    <include file="tenant/changelog-tenant.xml"
             context="tenant_db AND !initial_switch"/>

    <!-- Module-specific changelogs — add new modules at the END -->
    <include file="db/changelog/tenant/module/loan/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>
    <include file="db/changelog/tenant/module/investor/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>
    <include file="db/changelog/tenant/module/savings/parts/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>

    <!-- Custom extension point -->
    <includeAll path="db/custom-changelog" errorIfMissingOrEmpty="false"
                context="tenant_db AND !initial_switch AND custom_changelog"/>

    <!-- Additional modules -->
    <include file="/db/changelog/tenant/module/progressiveloan/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>
    <include file="db/changelog/tenant/module/loanorigination/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>
    <include file="db/changelog/tenant/module/command/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>
    <include file="db/changelog/tenant/module/workingcapitalloan/module-changelog-master.xml"
             context="tenant_db AND !initial_switch"/>

    <!-- Post-module finalization scripts -->
    <include file="tenant/final-changelog-tenant.xml"
             context="tenant_db AND !initial_switch"/>
</databaseChangeLog>
```

<Warning>
New modules must be added at the **end** of the module list to preserve
auto-increment IDs for all previously migrated data. Inserting a module in the
middle shifts IDs and will corrupt existing installations.
</Warning>

## Core Tenant Changeset Sequence

`fineract-provider/src/main/resources/db/changelog/tenant/parts/` contains 236
sequential XML changesets numbered `0001` through `0236`:

```
0001_initial_schema.xml        → base tables (loans, savings, clients, offices, …)
0002_initial_data.xml          → seed data (permission codes, product templates)
...
0236_<description>.xml         → most recent core schema change
```

## Database Context Switching

Liquibase contexts (`mysql`, `postgresql`) select DB-specific SQL fragments:

```xml
<changeSet id="..." author="..." context="mysql">
    <sql>ALTER TABLE m_loan ADD COLUMN ...</sql>
</changeSet>

<changeSet id="..." author="..." context="postgresql">
    <sql>ALTER TABLE m_loan ADD COLUMN ...</sql>
</changeSet>
```

The master changelog defines property aliases so that shared changesets can use
`${uuid}` and `${current_date}` tokens that expand to the correct function for
each database:

```xml
<property name="uuid" value="uuid()"             context="mysql"/>
<property name="uuid" value="uuid_generate_v4()" context="postgresql"/>
```

## Custom Changelog Extension Point

```xml
<includeAll path="db/custom-changelog" errorIfMissingOrEmpty="false"
            context="tenant_db AND !initial_switch AND custom_changelog"/>
```

Place custom `.xml` changeset files in
`fineract-provider/src/main/resources/db/custom-changelog/`. They are applied
only when the `custom_changelog` context is active. Use this for:

- Additional tables for custom modules (see [Build System — Custom Extension Pattern](/deployment/build-system))
- Data migrations for custom integrations
- Tenant-specific seed data

## Writing a New Changeset

### Naming Convention

```
<module>/src/main/resources/db/changelog/tenant/module/<module>/parts/
  NNNN_<short_description>.xml
```

Where `NNNN` is the next sequential number for that module's changelog.

### Required Attributes

```xml
<changeSet id="<module>-NNNN" author="<github-username>">
    <!-- context is optional; omit for DB-agnostic SQL -->
    <!-- or use context="mysql" / context="postgresql" for DB-specific -->
    <preConditions onFail="MARK_RAN">
        <not><tableExists tableName="my_new_table"/></not>
    </preConditions>
    <createTable tableName="my_new_table">
        <column name="id" type="BIGINT" autoIncrement="true">
            <constraints primaryKey="true" nullable="false"/>
        </column>
        <column name="name" type="VARCHAR(100)">
            <constraints nullable="false"/>
        </column>
    </createTable>
</changeSet>
```

**Rules:**
- `id` must be unique across the entire changelog (use a module-prefixed pattern).
- Always include `author`.
- Use `<preConditions onFail="MARK_RAN">` for idempotent DDL to protect
  against re-runs on already-migrated databases.
- Add DB-specific changesets in pairs (one with `context="mysql"`, one with
  `context="postgresql"`) when the SQL differs between databases.
- Register the new XML file in the module's `module-changelog-master.xml`.

## Total Scale

The full Fineract changelog tree (as of the current codebase snapshot) contains:

- **236** sequential core tenant parts (0001–0236)

- **11** tenant-store parts
- Multiple parts across loan, savings, command, progressive-loan, and other modules
